## The Execution Plan (Phase-by-Phase)

### Phase 1: Database Integrity & Final Schema Consolidation 🛢️
**Goal:** Guarantee a fully functional, secure, and relational backend that natively supports all features without manual fallback data.
- **Merge Schemas:** Combine schemas v1, v2, and v3 into a single, cohesive `garageos_schema_final.sql`.
- **Implement Row Level Security (RLS):** For absolute production readiness, we will write **strict `CREATE POLICY` statements** for `customers`, `vehicles`, `staff_profiles`, `work_orders`, etc., ensuring Owners, Managers, Technicians, and Customers can only read/mutate the data that is scoped to them.
- **Fix the 404 Error:** Ensure `staff_profiles` is properly created and exposed to the Supabase PostgREST API with proper permissions.

### Phase 2: Absolute Eradication of Mocks (Dynamic Data Bindings) 🔄
**Goal:** Scrub the codebase entirely of hardcoded variables, `MOCK_` arrays, and `DEMO_` datasets. Every single stat, row, and option must be generated by the database.
- **Staff Scanner Base:** Remove `DEMO_VEHICLES` from `app/staff/scan/page.tsx` and ensure it gracefully handles empty DB states without supplementing fake data.
- **Admin Garage Map:** Replace the hardcoded `BAYS` array with dynamic queries looking at `work_orders` and their assigned bays.
- **Analytics & Reports:** Strip hardcoded graphs and replace Recharts data sources with aggregated SQL derivations (e.g., total revenue per week from `invoices`).
- **Inventory Settings:** Scrub the static `MOCK_INVENTORY` list and replace it with real-time Supabase fetches. Ensure adding, editing, and deleting inventory items synchronizes deeply with the live database. 

### Phase 3: Elevating the "Scanner-to-3D" Pipeline 📷🚗
**Goal:** Implement the complex, original flow where staff scan vehicles to generate authentic 3D specs instead of a mocked representation.
- **Scanner UI to Implementation:** In `staff/scan/page.tsx`, map the QR flow to open an authenticated camera/upload flow that actually processes vehicle traits.
- **Color & Dimension Extraction:** We will refine the process where uploading an intake photo updates the `vehicles.color` and specific dimension metadata in Supabase.
- **3D React Fiber Integration:** Verify that `VehicleModelViewer` accurately reflects the precise color and dimensions retrieved from the actual database mapping, rather than arbitrary test defaults.

### Phase 4: Route Audits & Endpoint Checks 🛡️
**Goal:** Achieve 100% route safety with impenetrable gating and performative loads.
- **API Boundary Testing:** Verify that customer `auth.uid()` cannot guess and fetch invoice records of other customers.
- **Strict Role-Based Routing:** Ensure `PermissionGuard` actively ejects staff from admin routes, customers from staff routes, and secures APIs correctly.
- **Performance Constraints:** Implement caching strategies (via React Query/SWR or Next.js `fetch` `revalidate`) to prevent unneeded network spamming on high-traffic pages like the Admin Dashboard and Staff Jobs boards.

### Phase 5: Production Build Sequence 🚀
**Goal:** Execute continuous build checks and confirm the platform can be statically output without Next.js compiler panic.
- Analyze typescript strictness across all the freshly dynamic data (fixing implicit `any`s).
- Verify styling consistency (checking for CSS snapping/collapsing elements).
- Execute a final `npm run build` safely.
